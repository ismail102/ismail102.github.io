2 Tokenization

Convert text into tokens that models can process

from transformers import AutoTokenizer
import torch

tokenizer = AutoTokenizer.from_pretrained(
    "path/to/tokenizer", 
    do_lower_case=True
)

def tokenize(comment):
    encoded_dict = tokenizer.encode_plus(
        str(comment),
        add_special_tokens=True,
        max_length=512,
        pad_to_max_length=True,
        return_attention_mask=True,
        return_tensors='pt',
        truncation=True
    )
    
    return {
        "input_ids": encoded_dict['input_ids'],
        "mask": encoded_dict['attention_mask'],
        "token_type_ids": encoded_dict['token_type_ids']
    }

Model Initialization

Text Classification Model

Define a custom BERT-based classification model

from transformers import AutoModel
import torch.nn as nn

class TextClassification(nn.Module):
    def __init__(self, n_classes, dropout, model_ckpt):
        super(TextClassification, self).__init__()
        self.bert = AutoModel.from_pretrained(model_ckpt)
        self.dropout = nn.Dropout(dropout)
        self.out = nn.Linear(768, n_classes)
    
    def forward(self, ids, mask, token_type_ids):
        pooledOut = self.bert(
            ids, 
            attention_mask=mask,
            token_type_ids=token_type_ids
        )
        dropOut = self.dropout(pooledOut[1])
        output = self.out(dropOut)
        return output

# Initialize model
model = TextClassification(
    n_classes=5, 
    dropout=0.9, 
    model_ckpt="bert-base-uncased"
)

Prediction Function

Generate predictions from the trained model

def Classification(ids, mask, type_ids, device):
    # Move tensors to device
    ids = ids.to(device, dtype=torch.long)
    mask = mask.to(device, dtype=torch.long)
    token_type_ids = type_ids.to(device, dtype=torch.long)
    
    # Get model predictions
    outputs = model(
        ids=ids, 
        mask=mask, 
        token_type_ids=token_type_ids
    )
    
    # Apply sigmoid and convert to numpy
    result = torch.sigmoid(outputs)
    result = result.cpu().data.numpy()
    scores = result.tolist()
    
    return scores

§ 1 — Introduction & Course Overview

Two Core Memory Components in LLM Inference

When an LLM generates a response, it relies on two things stored in GPU memory.

ComponentDescriptionBehavior
Model Weights The billions of learned parameters (e.g., Llama, Qwen) Loaded once at startup; fixed regardless of request count
KV Cache Keys and values representing context from previous tokens Dynamic; every request has its own, grows with every new token
Memory Reality Check — Llama 3 70B
Model weights:          ~140 GB  (requires ≥ 2 × 80 GB GPUs just to load)
Standard production:     4 × 80 GB GPUs = 320 GB total
Remaining for KV cache: ~180 GB

One  8k-token request KV cache:  ~2.5 GB
One 32k-token request KV cache:  ~10 GB
→ You can serve ~18 long-context users in parallel
  (naive setups drop this to 2–3!)
Key Techniques You Will Learn
  • Quantization — shrinks model weights by storing them at lower numerical precision
  • PagedAttention — splits the KV cache into small, fixed-size blocks to eliminate memory waste
  • Continuous Batching — keeps the GPU busy by filling slots the moment a request finishes
  • Prefix Caching — reuses computed KV cache when multiple requests share the same prefix
  • GuideLLM — benchmarks your deployment under realistic load
  • LM-Eval — evaluates model quality on standardized benchmarks

§ 3 — Inference and Memory Fundamentals

The Three-Layer Inference Stack

┌─────────────────────────────┐
│         The Model           │  ← Learned parameters (Llama, Qwen, etc.)
├─────────────────────────────┤
│      Inference Server       │  ← vLLM: loads model, manages requests,
│                             │    applies all optimizations
├─────────────────────────────┤
│    Hardware Accelerator     │  ← GPU: does the heavy numerical lifting
└─────────────────────────────┘
Key insight: PyTorch works fine for a notebook, but once you need to serve many users simultaneously, the inference server becomes essential.

Autoregressive Generation & the KV Cache

LLMs generate one token at a time. A 500-token answer requires 500 full forward passes through the entire model.

Input:  "The quick brown"
Step 1: Model predicts → "fox"   → Input becomes "The quick brown fox"
Step 2: Model predicts → "jumps" → ...continues until <eos>
Why the KV Cache Exists

To generate token #5, you need K and V of tokens 1–4. But those haven't changed! The KV cache stores them so they don't need to be recomputed.

Step 1: Generate token 4 ("fox")   → Compute K4, V4 → SAVE to cache
Step 2: Generate token 5 ("jumps") → Only compute K5, V5 (new)
                                     Retrieve K1–K4, V1–V4 from CACHE ✓
# Savings multiply by N transformer layers.
KV Cache Size — Llama 3 70B
Formula: 2 × num_layers × num_KV_heads × head_dim × bytes_per_element

Llama 3 70B:  2 × 80 × 8 × 128 × 2 bytes = ~320 KB per token

At different context lengths:
   2,000 tokens  →   ~640 MB
   8,000 tokens  →   ~2.5 GB
  32,000 tokens  →   ~10 GB
 128,000 tokens  →   ~40 GB  ← Nearly 1/3 of model size, per user!

Serving 10 concurrent long-context users = 400+ GB of KV cache alone.

GPU Memory Hierarchy

┌──────────────────────────────────────────────────────┐
│  SRAM (On-chip, ~20 MB, ~19 TB/s bandwidth)          │  ← Tensor Cores read from here
├──────────────────────────────────────────────────────┤
│  HBM / VRAM (GPU card, 40–80+ GB, ~1.5 TB/s)        │  ← Model weights + KV Cache live here
├──────────────────────────────────────────────────────┤
│  CPU DRAM (Host machine, 16 GB – 1 TB+, ~12 GB/s)   │  ← Very slow relative to GPU
└──────────────────────────────────────────────────────┘
Two things govern inference speed: (1) how fast data moves HBM → SRAM, and (2) how fast Tensor Cores compute once data arrives. Every optimization in this guide comes back to: move less data, move it more efficiently, manage memory better.

Quantization — Numerical Formats

Instead of storing weights at full precision (e.g., 3.14159265...), store them at lower precision (e.g., 3.14). Moving down this table = trading precision for size.

FormatBitsDescription
FP3232Full precision; huge range, very precise
BF1616Brain Float 16; same range as FP32, less precision — standard for LLM release
FP1616Floating Point 16; smaller range than BF16
FP888-bit float; supported on Hopper (H100) GPUs
INT888-bit integer; whole numbers like -127 to 127
INT444-bit integer; most aggressive, highest compression
What Gets Quantized?
Quantized:       Q, K, V, O projections (Self-Attention)
                 gate, up, down projections (Feed-Forward Network)

NOT quantized:   Embedding layer (input)
                 LM Head (output)
                 Layer normalization layers
Two Quantization Schemes
# W8A16 — Weight-Only (8-bit weights, 16-bit activations)
Weights stored INT8 → less data moves HBM → SRAM → faster data movement
Math stays BF16 → no Tensor Core speedup, but memory savings are real

# W8A8 — Weight + Activation (both 8-bit)
Less data moves HBM → SRAM (faster)
Math runs on dedicated FP8/INT8 Tensor Cores (more ops/sec)
Requirement: Newer GPUs (Hopper for FP8, Ampere for INT8)
Quantization Impact — Llama 4 Scout (109B Parameters)
BF16 baseline:  109B × 2 bytes = ~220 GB  → needs 3× 80 GB GPUs
FP8 / INT8:     109B × 1 byte  = ~109 GB  → needs 2× 80 GB GPUs  (50% reduction)
INT4 / FP4:     109B × 0.5 byte =  ~55 GB  → needs 1× 80 GB GPU   (75% reduction!)
Real-World Gains — FP16 vs FP8, Llama 3 70B on 2× H100s
MetricFP16 BaselineFP8 QuantizedImprovement
Throughput~158 tokens/sec~474 tokens/sec3× faster
TTFT at high load>30,000 ms~4,800 ms67× faster

4 Compare Model Sizes 

import os

def get_model_size_gb(path):
    total = sum(os.path.getsize(os.path.join(path, f))
               for f in os.listdir(path)
               if os.path.isfile(os.path.join(path, f)))
    return total / (1024 ** 3)

orig = get_model_size_gb(base_model_path)
quant = get_model_size_gb(quantized_model_path)
print(f"Original  (BF16):  {orig:.2f} GB")
print(f"Quantized (W4A16): {quant:.2f} GB")
print(f"Size reduction:    {(1 - quant/orig)*100:.1f}%")
# Qwen3-0.6B: ~1.2 GB → ~0.7 GB → ~42% reduction
Why 42% instead of the theoretical 75%? Only the linear layer weights are quantized. The LM head, embeddings, and norm layers stay at higher precision. In larger models (70B+), linear weights dominate, so you get much closer to the 4× theoretical.

5 Verify Quality with Perplexity

Perplexity measures how well a model predicts text — lower is better. If quantization degraded the model, perplexity rises noticeably.

def calculate_perplexity(model, tokenizer, text, max_length=512, stride=256):
    encodings   = tokenizer(text, return_tensors="pt")
    input_ids   = encodings.input_ids
    nlls, prev  = [], 0

    for begin in range(0, input_ids.size(1), stride):
        end        = min(begin + max_length, input_ids.size(1))
        trg_len    = end - prev
        chunk      = input_ids[:, begin:end]
        labels     = chunk.clone()
        labels[:, :-trg_len] = -100
        with torch.no_grad():
            out = model(chunk, labels=labels)
            nlls.append(out.loss * trg_len)
        prev = end
        if end == input_ids.size(1): break

    return torch.exp(torch.stack(nlls).sum() / end).item()

# Qwen3-0.6B result:
# Base perplexity:      32.79
# Quantized perplexity: 35.48  (~8% increase, 42% size reduction — great tradeoff)

§ 6 — Serving with vLLM — Core Concepts

Continuous Batching

Static batching waits for every request in a batch to finish before starting new ones, leaving GPU slots idle. Continuous batching fills slots the instant any request completes.

# Static Batching — broken for LLMs
Time →   T1  T2  T3  T4  T5  T6  T7  T8
Req 1:   ████████████████████              (done T6)
Req 2:   ██████████████████████████████   (done T8)
Req 3:   ███████████                      (done T5)
         ↑ All 4 start together → Batch 2 can't start until ALL finish at T8
           Slots for Req 3 sit IDLE from T5 to T8

# Continuous Batching — vLLM's approach
Time →   T1  T2  T3  T4  T5  T6  T7  T8
Req 1:   ████████████████████              (done T6)
Req 3:   ███████████                      (done T5)
Req 5:              ██████████████████    ← starts at T5 when Req 3 finishes!
Req 6:                    ████████████   ← starts at T6 when Req 1 finishes!

The GPU batch is NEVER IDLE — scheduler works at the token level.

PagedAttention

Inspired by OS virtual memory paging. Breaks the KV cache into small, fixed-size blocks that can live anywhere in GPU memory — no pre-allocation, no wasted slots, no fragmentation.

# Old approach: one contiguous block pre-allocated per request
┌──────────────────────────────────────────────────────────┐
│ Request A │▓▓▓▓▓▓░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░│
│  (2048)  │ used │       NEVER USED (fragmentation)  │
└──────────────────────────────────────────────────────────┘
Result: Only 20–40% of KV cache memory held real tokens!

# PagedAttention: non-contiguous blocks anywhere in memory
Physical GPU Memory:
┌────────┬────────┬────────┬────────┬────────┬────────┐
│Block 0 │Block 1 │Block 2 │Block 3 │Block 4 │Block 5 │
│(free)  │(Req B) │(free)  │(Req A) │(Req B) │(free)  │
└────────┴────────┴────────┴────────┴────────┴────────┘

Block Table for Request A:      Block Table for Request B:
  Logical → Physical              Logical → Physical
  0-3     → Block 3               0-3     → Block 1
  4-7     → Block 6               4-7     → Block 4
  (allocated only as needed)      (allocated only as needed)

Prefix Caching

When multiple requests share the same prefix (e.g., the same system prompt), the KV cache for that prefix only needs to be computed once.

# Without Prefix Caching
User 1: "[500-token system prompt] → Question 1"
         └── Full KV cache computed for 500-token prefix
User 2: "[same 500-token system prompt] → Question 2"
         └── Full KV cache computed AGAIN  (wasted!)

# With Prefix Caching
User 1: Full computation → CACHED
User 2: Cache hit! Skip 500 tokens → only compute new question
User 3: Cache hit! Skip 500 tokens → only compute new question

At 75% cache hit rate → throughput is ~4× higher
Also Applies To
  • Multi-turn conversations (round 2 includes all of round 1's content)
  • Few-shot examples shared across users
  • Shared RAG context (retrieved documents)

Monitoring vLLM Metrics

vLLM exposes a Prometheus-compatible endpoint at /metrics for observability.

import requests, re

def get_vllm_metrics(base_url="http://localhost:8000"):
    text    = requests.get(f"{base_url}/metrics").text
    metrics = {}
    patterns = {
        "running_requests":    r'vllm:num_requests_running\s+([\d.]+)',
        "gpu_cache_usage_pct": r'vllm:gpu_cache_usage_perc\s+([\d.]+)',
        "total_prompt_tokens": r'vllm:prompt_tokens_total\s+([\d.]+)',
    }
    for key, pat in patterns.items():
        m = re.search(pat, text)
        if m: metrics[key] = float(m.group(1))
    return metrics

print(get_vllm_metrics())
# {'running_requests': 0.0, 'gpu_cache_usage_pct': 2.3, 'total_prompt_tokens': 847.0}

    --target http://localhost:8000 \
    --profile synchronous \
    --max-requests 100 \
    --data-type emulated \
    --prompt-tokens 32 \

Always look at p95 and p99, not just the mean. If p99 is much higher than the mean, 1 in 100 users waits far longer — users notice even when the average looks fine.

Key Formulas & Rules of Thumb

KV Cache per token (Llama 3 70B):
  2 × 80 layers × 8 KV heads × 128 head_dim × 2 bytes = 320 KB

Model size (BF16):
  num_parameters × 2 bytes = size in bytes

After W4A16 quantization (linear layers only):
  Overall model reduction ≈ 40–75%  (larger models → closer to theoretical)

Calibration samples:
  256 samples = solid default; diminishing returns past ~512

GPU memory rule of thumb for Llama 70B:
  Minimum: 2× 80 GB GPUs (load model)
  Production: 4× 80 GB GPUs (leaves room for KV cache)

From Classical Machine Learning to Agentic LLMs

1 Why This Matters

Why modern AI is a systems problem, not just a modeling problem

Core Motivation:
  • Beyond Accuracy: Real-world AI must reason, adapt, and interact with tools
  • System-Level Thinking: Performance depends on architecture, data, and evaluation
  • Reliability: AI must behave robustly under uncertainty and distribution shifts
  • Alignment: Systems must remain controllable and human-centered
Key Insight: As models scale, system design becomes the primary bottleneck.

2 Classical Machine Learning Still Matters

Why foundational ML remains essential in LLM-centric systems

Where Classical ML Fits:
  • Structured Signals: Tabular, temporal, and metadata-driven learning
  • Baselines: Strong reference points for evaluating LLM gains
  • Monitoring: Drift detection, anomaly detection, and calibration
  • Efficiency: Lightweight models for cost-sensitive components
Lesson: LLMs augment classical ML—they rarely replace it entirely.

3 Multimodal Learning: Beyond Text

Aligning multiple modalities for unified reasoning

Key Concepts:
  • Representation Alignment: Mapping text, vision, and signals into shared spaces
  • Cross-Modal Reasoning: Using one modality to disambiguate another
  • Vision-Language Models: Joint perception and language understanding
  • Real-World Inputs: Video, sensors, and structured context
Perspective: Intelligence emerges when modalities reinforce each other.

4 Large Language Models as System Components

Treating LLMs as interfaces, planners, and controllers

Practical Building Blocks:
  • Instruction Tuning: Aligning model behavior with task intent
  • Post-Training: PPO, DPO, GRPO for preference alignment
  • PEFT: Efficient adaptation without full retraining
  • RAG: Grounding generation in external knowledge
Observation: Many LLM failures are architectural, not linguistic.

5 Reasoning and Agentic Behavior

Designing AI systems that plan, act, and adapt

Agent Design Elements:
  • Planning: Multi-step decision making over long horizons
  • Tool Use: APIs, search, code execution, and external actions
  • Memory: Persistent and episodic context
  • Control: Constraining behavior through policies and rewards
Reality: Agentic behavior is engineered—not emergent magic.

6 Safety, Robustness, and Evaluation

Ensuring trustworthy behavior in autonomous systems

Evaluation Focus:
  • Failure Modes: Identifying where and why systems break
  • Distribution Shift: Robustness beyond training data
  • Adversarial Inputs: Stress-testing unsafe behavior
  • Human-in-the-Loop: Oversight and corrective feedback
Principle: Safety must be designed into the system, not added later.

7 How Everything Connects

Viewing AI as an integrated system

System Interactions:
  • Classical ML: Grounding and structure
  • Multimodal Models: Rich perception
  • LLMs: Language and reasoning
  • Agents: Long-horizon task execution
  • Evaluation: Trust and reliability
Takeaway: Progress happens between components, not inside one model.

8 Final Thoughts

Where modern AI research is heading

Closing Reflections:
  • Bigger is Not Enough: Scaling alone does not ensure reliability
  • Systems Matter: Architecture, data, and evaluation define success
  • Agentic AI: The future lies in planning, memory, and control

Federated Learning

1 What is Federated Learning?

Distributed machine learning approach that preserves privacy

Privacy-Preserving: Train models on distributed data without centralizing it
Key Principles:
  • Data Privacy: Data never leaves the device
  • Decentralized Training: Models trained locally on devices
  • Aggregation: Central server aggregates model updates
  • Communication Efficiency: Minimize data transfer

2 Federated Averaging Algorithm

The core algorithm for federated learning

def federated_averaging(global_model, client_models, client_weights):
    """
    Aggregate client models using weighted averaging
    
    Args:
        global_model: The current global model
        client_models: List of updated client models
        client_weights: Weight for each client (usually based on data size)
    """
    
    # Initialize aggregated parameters
    aggregated_params = {}
    
    for name, param in global_model.named_parameters():
        aggregated_params[name] = torch.zeros_like(param.data)
    
    # Weighted averaging
    for client_model, weight in zip(client_models, client_weights):
        for name, param in client_model.named_parameters():
            aggregated_params[name] += weight * param.data
    
    # Update global model
    for name, param in global_model.named_parameters():
        param.data = aggregated_params[name]
    
    return global_model

3 Challenges in Federated Learning

Understanding the limitations and ongoing research areas

Main Challenges:
  • Non-IID Data: Data distribution varies across clients
  • Communication Costs: Limited bandwidth and connectivity
  • System Heterogeneity: Different device capabilities
  • Privacy Attacks: Model inversion and membership inference
  • Byzantine Failures: Malicious or faulty clients
Note: Differential privacy and secure aggregation are common solutions
